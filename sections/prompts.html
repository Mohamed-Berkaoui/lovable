PRD → Lovable Prompts

Transform your structured PRD into precise prompts that Lovable can execute perfectly.

🕑 Estimated reading time: 14 minutes

How Lovable Interprets Prompts

Lovable's AI reads your prompt, breaks it down into tasks, and generates code for each task. It processes prompts sequentially and literally. This means:

  • Every sentence can result in a code decision.
  • Ambiguity leads to the AI choosing for you — and it might choose wrong.
  • Order matters: the AI processes from top to bottom.
  • Shorter, focused prompts yield better results than massive monolithic ones.

📚 Key Concept

Think of Lovable's AI as a very fast, very literal junior developer. It will do exactly what you say — but it won't read between the lines. If you leave something out, it won't be built.

Prompt Engineering Basics

Prompt engineering is the skill of writing instructions that produce the best possible output from an AI. Here are the fundamental principles:

1. Be Specific

Instead of "add a navbar", say:

Prompt Add a fixed horizontal navigation bar at the top of every page. It should contain: - Logo on the left (link to home page) - Navigation links: Home, Products, Cart, Account - A shopping cart icon with item count badge on the right - On mobile: collapse to a hamburger menu

2. One Task Per Prompt

Avoid sending a 500-word prompt that asks for 10 things at once. Break it down into individual prompts, each focused on a single feature or page.

3. Use Structured Formatting

Use bullet points, numbered lists, and clear headings. The AI parses structured text much better than long paragraphs.

4. Include Constraints

Tell the AI what not to do. For example: "Do NOT add pagination. Show all products on one page."

5. Reference Previous Context

If you've already built a page, reference it: "On the existing product detail page, add a reviews section below the specs table."

Prompt Structure Template

Use this template for every prompt you send to Lovable:

Template ## [Task Title] ### What to build: [Clear description of the feature or page] ### Requirements: - [Requirement 1] - [Requirement 2] - [Requirement 3] ### Layout / UI: - [Position, size, style details] ### Behaviour: - [What happens on click, load, submit, etc.] ### Constraints: - [What NOT to do] - [Performance / accessibility requirements] ### Data: - [What data is needed, where it comes from]

Example: Page Generation Prompts

Prompt 1: Home Page

Prompt ## Build the Home Page ### What to build: The main landing page for TechMart e-commerce store. ### Requirements: - Hero section with a large banner image, headline, and CTA button - Featured products section showing 8 products in a 4-column grid - Category navigation section with icon cards (Phones, Laptops, Accessories, Audio) - Footer with links, social icons, and copyright ### Layout / UI: - Full-width hero with gradient overlay on the image - Product cards: image, name, price, "Add to Cart" button - Category cards: icon + label, hover effect with scale ### Behaviour: - CTA button scrolls to featured products - Product cards link to product detail page - Category cards link to filtered category page ### Constraints: - Do NOT add a sidebar - Maximum 8 featured products - Mobile responsive: 2-column grid on tablet, 1-column on phone ### Data: - Products from Supabase "products" table - Categories are hardcoded for now

Prompt 2: Product Detail Page

Prompt ## Build the Product Detail Page ### What to build: A detail page for a single product, accessible via /product/:id. ### Requirements: - Image gallery (main image + thumbnail strip) - Product name, price, description - Specifications table (key-value pairs) - Quantity selector (1-10) - "Add to Cart" button - Related products section (4 items from same category) ### Layout / UI: - Two-column layout: images left, details right - On mobile: stack vertically (images on top) - Specs in a striped table - Related products in a horizontal scroll on mobile ### Behaviour: - Click thumbnail to change main image - "Add to Cart" adds to local cart state and shows toast - Related products link to their own detail pages ### Data: - Product data from Supabase by ID - Related products: query same category, exclude current

Example: Feature Prompts

Adding Authentication

Prompt ## Add User Authentication ### What to build: Email + Google OAuth authentication using Supabase Auth. ### Requirements: - Sign Up page: email, password, confirm password, Google button - Sign In page: email, password, Google button - "Forgot Password" link that triggers Supabase reset email - Protected routes: Cart and Account require login - Show user avatar + name in navbar when logged in ### Behaviour: - After sign up, redirect to home with success toast - After sign in, redirect to previous page or home - Unauthenticated users clicking Cart → redirect to Sign In - Sign out button in account dropdown ### Constraints: - Do NOT add phone number auth - Do NOT add Magic Link auth - Passwords must be minimum 8 characters

Adding a Shopping Cart

Prompt ## Build Shopping Cart Feature ### What to build: A client-side shopping cart stored in localStorage. ### Requirements: - Cart icon in navbar with badge showing item count - Cart page showing all added items - Each item shows: image, name, price, quantity, subtotal - Quantity controls: +/- buttons, minimum 1 - Remove item button - Cart total at the bottom - "Proceed to Checkout" button ### Behaviour: - Adding same product increments quantity - Removing last item shows "Your cart is empty" message - Cart persists across page reloads (localStorage) - "Proceed to Checkout" goes to /checkout (requires auth) ### Constraints: - Do NOT integrate payments yet (that's a separate prompt) - Maximum 99 of any single item

💡 Pro Tip

Send prompts in a logical order: layout first, then data, then interactions, then edge cases. This mirrors how a human developer would build.

⚠️ Common Mistake

Don't ask Lovable to "make it look good" or "use best practices". These are too vague. Instead, specify exact colours, spacing, fonts, and layout rules.

Section Summary

  1. Lovable reads prompts literally — be specific and structured.
  2. Use the prompt template: Task → Requirements → Layout → Behaviour → Constraints → Data.
  3. One prompt per feature or page for best results.
  4. Include explicit constraints (what NOT to do).
  5. Send prompts in logical build order: layout → data → interactions → edge cases.
